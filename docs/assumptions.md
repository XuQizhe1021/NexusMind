# NexusMind 工程假设与取舍

## A1 Provider 范围（Phase 1）

- 假设：首阶段仅稳定支持 OpenAI，Claude/Gemini 保留配置入口
- 原因：先闭环最小可运行链路，降低接入面复杂度
- 回退：若 OpenAI 不可用，可切换到 mock provider 维持演示能力

## A2 API Key 加密实现

- 假设：基于 `chrome.runtime.id` + PBKDF2 + AES-GCM 完成本地加密
- 原因：在无后端密钥托管前提下，保证“非明文落盘”底线
- 回退：若后续接入账户系统，升级为服务端短期令牌，不再本地保存长期密钥

## A3 页面文本提取策略

- 假设：Phase 1 使用 `document.body.innerText` 作为问答上下文
- 原因：实现快、覆盖广，便于验证端到端链路
- 回退：若遇到站点噪声高，在 Phase 2 增加可读性过滤与分块检索

## A4 商业化实现时点

- 假设：订阅校验、计量、风控在 Phase 5 完整实现
- 原因：先完成核心体验，避免前期过度工程化
- 回退：若试点期已有付费需求，可提前最小化实现月度额度扣减

## A5 图谱抽取策略（Phase 2）

- 假设：Phase 2 采用启发式词元抽取（中文短词 + 英文词）构建实体候选
- 原因：在无额外模型推理成本前提下，快速建立跨页图谱基础能力
- 回退：若噪声实体偏高，升级为“LLM 抽取 + 规则校验”的双阶段流水线

## A6 图谱归一键策略（Phase 2）

- 假设：实体归一以 `canonicalKey`（小写 + 去噪）作为主键
- 原因：确保跨页大小写差异可合并，避免重复实体膨胀
- 回退：若同名异义冲突明显，引入“站点域 + 上下文向量”联合键

## A7 图谱回滚策略（Phase 2）

- 假设：页面重复收录时先回收旧引用，再写入新索引
- 原因：保证单页重建可回滚且结果幂等，减少脏关系残留
- 回退：若出现写入中断，提供“一键清空图谱”恢复路径

## A8 页面重写焦点区识别策略（Phase 3）

- 假设：优先按 `article/main/[role=main]/.content` 识别正文容器，未命中时退化为“文本最多的 body 一级子节点”
- 原因：在覆盖常见内容站点的同时，降低策略复杂度并保证性能上限
- 回退：若站点误判明显，可按域名增加站点级重写规则或关闭该站点默认意图

## A9 页面重写回滚模型（Phase 3）

- 假设：重写会话记录“插入面板、节点移动、样式修改、元素隐藏”四类可逆变更
- 原因：以最小状态记录换取稳定回滚，不依赖整页 HTML 快照以减少内存占用
- 回退：若检测到不可预期 DOM 并发改写，可触发完整刷新恢复页面原始状态

## A10 SPA 路由重写隔离（Phase 3）

- 假设：通过监听 `pushState/replaceState/popstate/hashchange` 判定路由变化并自动回滚
- 原因：确保旧路由的重写状态不会污染新路由页面，降低“残留样式/结构”风险
- 回退：若站点使用非标准路由机制，可手动点击“一键还原”并在后续增加专用兼容适配

## A11 图谱问答证据构建策略（Phase 4）

- 假设：问题先做轻量关键词抽取，再在图谱中聚合实体/关系并回收关联页面作为证据来源
- 原因：在不引入额外向量数据库的前提下，先保证跨页检索可用且可解释
- 回退：若召回率不足，升级为“关键词 + 向量检索”混合召回并引入重排

## A12 引用定位策略（Phase 4）

- 假设：正文定位采用浏览器 `window.find` + 片段匹配，命中后执行局部高亮
- 原因：兼容成本低、可快速覆盖主流页面结构且支持清理回退
- 回退：若复杂站点命中率不足，补充基于 TextNode 索引的精确定位器

## A13 流式中断控制策略（Phase 4）

- 假设：sidepanel 与 background 通过长连接传输增量分片，后台使用 `AbortController` 终止请求
- 原因：在 MV3 架构下可稳定实现增量渲染与可中断请求，避免 UI 阻塞
- 回退：若连接异常，可降级为非流式单次返回并保留取消态提示

## A14 订阅校验来源（Phase 5）

- 假设：在账户系统未上线前，订阅校验先采用本地 Token 规则（`nm_sub_` 前缀）驱动状态机
- 原因：保证商业化链路可端到端演练，同时避免引入未落地后端依赖
- 回退：接入真实订阅服务后，以服务端签名校验替换本地规则

## A15 计量扣减时点（Phase 5）

- 假设：仅在问答会话成功完成后扣减调用次数，取消/失败不扣减
- 原因：计费口径清晰且与“每次问答完成记一次调用”一致
- 回退：若后续策略改为“请求即计费”，可在调用前改为预扣并在失败时回补

## A16 退款与风控策略（Phase 5）

- 假设：退款处理后立即回收订阅权益；风控命中后阻断调用并保留审计轨迹
- 原因：先保证风险可控与状态可追踪，满足审计最小闭环
- 回退：接入客服审核系统后，改为“人工复核 + 分级解封”流程

## A17 流式断流重连策略（Phase 6）

- 假设：对可重试错误执行最多 2 次重连，不做跨尝试增量续传
- 原因：OpenAI SSE 不提供稳定续传锚点，优先保证实现可控和一致性
- 回退：若重连成功率不足，升级为“可恢复游标 + 断点续流”协议

## A18 计量并发一致性口径（Phase 6）

- 假设：按 requestId 做调用幂等去重，重复完成事件只记审计不重复扣减
- 原因：服务工作线程重启/重连重放时可避免重复计费
- 回退：若并发规模继续提升，升级为持久化事务队列 + CAS 锁

## A19 兼容性回归执行方式（Phase 6）

- 假设：先建立固定回归集并执行 Chrome/Edge 手工批量回归，自动化回归后置
- 原因：当前项目尚未引入跨浏览器自动化框架，先保证覆盖口径稳定
- 回退：Phase 7 引入 Playwright 矩阵化自动回归并接入 CI 门禁
